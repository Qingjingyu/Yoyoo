# OpenClaw 两周体验：AI 员工化实践

> **调研日期**: 2026-02-04
> **来源**: 用户提供长文（刘小排）
> **调研目标**: 提炼 OpenClaw 深度使用两周的经验，评估“AI 员工化”对团队的影响与落地建议。

## 1. 结论摘要
- **用 AI 代替扩招**：暂停招人，把 OpenClaw 视作核心生产力。
- **权限 + SOP 即上限**：给足 root 权限 + 清晰 SOP，执行力≈“最聪明人+电脑”。
- **环境优先级**：海外 Linux VPS（大带宽、root 权限）优于本地 Mac Mini；Mac 专属技能非刚需。
- **模型选型**：综合最强 Opus-4.5；编程专项 GPT-5.2-Codex；差距明显，简单任务不显现。
- **成本打法**：ChatGPT Pro 订阅可支撑两台高负荷；Opus-4.5 参考“不会封号的 Claude Code”方案。
- **浏览器自动化**：严肃自动化首选 Browser-use，优于 Playwright MCP/Agent Browser。
- **一键安装是幻觉**：需像高级工程师入职，配置环境+融入流程，不可能“一键装好”。
- **失败常因不会用**：无任务/不会设计 SOP/技能不足导致“觉得没用”。

## 2. 使用体验要点
- 把 OpenClaw 当“人”用：SOP 驱动全流程（拉代码、审查、白盒测试、多角色用例、回报）。
- 权限越大越省心：root + 高带宽服务器提升稳定性与执行速度。
- 浏览器自动化：Browser-use 适合严肃任务；内置 MCP/Agent 浏览器不如它。
- 模型差异：Opus-4.5 通用最佳，GPT-5.2-Codex 编程最佳，其他模型差距大。

## 3. 推荐实践
1) **基础设施**：优先海外 Linux VPS，开放 root；准备高带宽，端口和安全组放行所需范围。
2) **SOP 设计**：为常见任务写清步骤与验收（拉代码、测试矩阵、上线流程）。
3) **权限管理**：最小阻力但可审计；关键操作配套日志与回执。
4) **模型路由**：通用=Opus-4.5，编程=GPT-5.2-Codex；任务简单时也可降级节省成本。
5) **自动化工具**：浏览器自动化用 Browser-use；API/CLI 任务直接走终端+脚本。
6) **期望管理**：不求“一键安装”，按“高级工程师入职”心态搭环境和流程。

## 4. 风险与对策
- **权限风险**：root 误操作 → 建立审计日志与备份，关键目录快照。
- **SOP 缺失**：输出不可控 → 先写 SOP，再放权；逐步放大任务粒度。
- **模型不稳/成本**：准备备选模型与限流策略，任务分级路由。
- **网络/带宽瓶颈**：监控与冗余节点，必要时多地域切换。

## 5. 对 Yoyoo 的启示
- 沿用“AI 员工化”思路：核心任务写 SOP、放权到 Agent，减少人力扩张。
- 重点补齐浏览器自动化链路（Browser-use）与模型路由策略（Opus-4.5 / GPT-5.2-Codex）。
- 基础设施以海外 Linux VPS 为主，优化带宽与权限，减少本地依赖。
